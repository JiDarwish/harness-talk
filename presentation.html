The inner harness is just a loop

while (true) {
    var response = chat(messages, tools);
    messages.add(assistant(response));

    if (response.toolCalls().isEmpty()) break;

    for (var call : response.toolCalls()) {
        messages.add(toolResult(call.name(), execute(call)));
    }
}

Four operations repeat

  1. Send conversation and tools to the model
  2. Parse text and tool calls
  3. Execute allowed tools
  4. Append tool results back into the conversation

Everything else in this talk is about improving the harness around those four operations.

Tools define the action space

A coding agent is not powerful because it can "think" about files. It becomes powerful when we expose actions:

  • Read a file
  • Write a file
  • Run a test
  • Run a codemod
  • Load a project skill
03
OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

Prompt, permissions, observability

The inner harness needs mundane engineering:

  • System prompt that says what kind of assistant this is
  • Path restrictions and tool permissions
  • Transcript of actions and results
  • Failures that are visible enough to debug

The boring parts are what make the exciting parts controllable.

OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

The wall

An inner harness can edit code, but it does not know your organization.

  • Repeats the same local mistake
  • Hand-edits transformations you already know how to automate
  • Violates architecture without noticing
  • Writes tests that prove the wrong behaviour

Outer harness: the job

Two goals:

  • Steer before action with context, conventions, and deterministic tools
  • Catch after action with sensors the agent can use to self-correct

This is where agentic coding becomes software engineering again.

AGENTS.md: feedforward inferential

It gives the model a compressed version of project judgment:

  • Domain rules
  • Architecture boundaries
  • Testing conventions
  • Preferred APIs
  • Forbidden shortcuts

It improves the first attempt, but it cannot prove the result.

04
OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

Skills: context on demand

Do not cram every rule into the first prompt. Expose focused guidance as a tool:

agent.registerTool(LoadSkill.SPEC,
    tu -> tu.input(LoadSkill.class).execute());

The agent asks for the detail when the task needs it.

04
OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

Codemod: feedforward computational

Some work should not be free-handed by a model.

agent.registerTool(RunCodemod.SPEC,
    tu -> tu.input(RunCodemod.class).execute());

If a transformation is known, encode it once and let the agent choose it.

04
OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

Guides do not verify

Feedforward controls reduce the chance of a bad first attempt. They do not answer:

  • Did the agent follow the rule?
  • Did the edit compile?
  • Did the architecture boundary hold?
  • Did the behaviour match the business rule?

That is why the next half of the talk is about sensors.

Linter sensor: feedback computational

agent.afterWriteHook = (filePath, content) -> {
    if (!filePath.endsWith(".java")) return null;

    var output = runResultUnwrapChecker(filePath);
    return output.failed()
        ? "LINTER FEEDBACK:\n" + output.text()
        : null;
};

The linter does not just fail. It teaches the agent how to repair.

LLM-as-judge: feedback inferential

    architectureReviewPrompt,

Keep quality left

Prefer deterministic controls when the rule is known:

  • Codemods for repeatable edits
  • Tests for behaviour you can specify
  • Linters for local constraints
  • Architecture tests for boundaries

The outer harness should make good work cheaper, not just make bad work louder.

The behaviour gap

All sensors can pass and the feature can still be wrong.

Waive the late fee if a book is returned within 1 hour of library opening (9 AM). The library is in US/Eastern.

The agent uses LocalDateTime.now(). The server runs in UTC.

A 9:30 AM Eastern return shows as 1:30 PM UTC. Fee charged. Tests pass because they share the same wrong assumption.

Why green tests can lie

An agent can write:

  • Code
  • Tests
  • Documentation
  • Explanations

If all four are generated from the same misunderstanding, consistency is not correctness.

06
OPENINGHARNESSINNEROUTERSENSORSGAPCLOSE

BDD becomes the engineer's most valuable skill

Agents can write code, tests, and docs. What they cannot do is imagine what should go wrong.

  • What happens at midnight when the timezone flips?
  • What happens when a member returns a book on a holiday?
  • What happens when two members borrow the last copy at the same time?

Thinking in edge cases, writing behaviour specifications, and encoding business intent as tests -- this is how engineers stay ahead of the gap.

The most valuable engineer is not the one who writes the most code. It is the one who specifies the most edge cases.

What changes for engineers

We stop asking "is the model good enough?" We start asking "is the system around it good enough?"

  • Narrow permissions
  • Repeatable tools
  • Executable conventions
  • Feedback designed for repair
  • Review where semantics are uncertain
